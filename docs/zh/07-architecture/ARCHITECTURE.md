## 系统概述

CDNProxy 是一个企业级的 CDN 和 API 代理服务，旨在解决在受限网络环境下访问公共 CDN 资源和 AI 服务 API 的问题。

### 核心功能

- **CDN 代理**：智能反向代理，支持硬盘缓存、访问控制、大文件流式传输
- **API 代理**：支持 AI 服务 API（OpenAI、Claude 等），支持 WebSocket、SSE 流式响应
- **管理后台**：Web 界面管理白名单、配置和统计信息
- **住宅 IP 代理**：支持通过住宅 IP 代理访问 AI API，提高成功率

### 架构层次说明

1. **HTTP 服务器层**：基于 Go 标准库 `net/http`，提供 HTTP/HTTPS 服务
2. **中间件层**：全局 panic 恢复、请求日志记录
3. **路由分发层**：根据路径分发到不同的处理器
4. **业务逻辑层**：CDN 代理、API 代理、管理后台
5. **数据存储层**：缓存（Redis）、配置存储（Redis）
6. **上游服务层**：目标 CDN 或 API 服务

## 核心模块

### 1. HTTP 服务器模块 (`main.go`)

**职责**：
- 初始化所有组件
- 设置 HTTP 服务器配置
- 注册路由
- 优雅关闭处理

**关键组件**：
```go
- panicRecoveryMiddleware: 全局 panic 恢复
- loggingMiddleware: 请求日志记录
- monitorResources(): 资源监控 goroutine
- setFDLimit(): 文件描述符限制设置
```

**超时配置**：
- `ReadHeaderTimeout`: 10 秒
- `ReadTimeout`: 10 分钟（支持长连接）
- `WriteTimeout`: 10 分钟（支持长连接）
- `IdleTimeout`: 60 秒

### 2. 代理处理器模块 (`proxy/handler.go`)

**职责**：
- 处理所有代理请求
- 区分 CDN 和 API 请求
- 访问控制检查
- 缓存管理
- 大文件流式传输

**核心流程**：
```
1. 接收请求
2. 获取信号量（并发控制）
3. 构建上游 URL
4. 判断请求类型（CDN vs API）
5. CDN 请求：
   - 访问控制检查
   - 缓存查询
   - 缓存未命中时请求上游
   - 缓存响应
6. API 请求：
   - 直接转发到 API 代理
```

**关键特性**：
- 并发控制：信号量限制最多 50 个并发请求
- 内存池：64KB 缓冲区池，减少内存分配
- 大文件处理：>1MB 文件流式传输，不缓存
- WebP 转换：可选的图片格式转换

### 3. AI API 代理模块 (`proxy/ai_api_proxy.go`)

**职责**：
- 代理 AI 服务的 API 请求
- 管理住宅 IP 代理
- 处理流式响应
- 代理健康检查和统计

**核心流程**：
```
1. 检测 API 类型（OpenAI/Claude/Gemini 等）
2. 获取最佳住宅 IP 代理
3. 创建带代理的 HTTP 客户端
4. 发送请求到上游
5. 流式传输响应（支持 context 取消）
6. 报告代理使用情况
```

**支持的 API 类型**：
- OpenAI (`api.openai.com`)
- Claude (`api.anthropic.com`)
- Gemini (`generativelanguage.googleapis.com`)
- Poe (`poe.com`)
- 其他（可通过环境变量配置）

### 4. API 代理模块 (`proxy/api_proxy.go`)

**职责**：
- 处理普通 API 请求
- WebSocket 代理
- SSE 流式响应处理
- 长连接支持

**核心功能**：
- **WebSocket 代理**：
  - 支持双向通信
  - 超时控制（5 分钟）
  - 并发限制（最多 10 个连接）
  
- **SSE 流式传输**：
  - 逐行读取和转发
  - 超时控制（10 分钟）
  - 支持 context 取消

- **普通 API 请求**：
  - 支持长超时（5 分钟）
  - 检查 context 取消
  - 错误处理和日志记录

### 5. 住宅 IP 代理管理器 (`proxy/residential_proxy_manager.go`)

**职责**：
- 管理多个住宅 IP 代理提供商
- 代理健康检查
- 代理选择策略
- 使用统计和成本优化

**支持的提供商**：
- Bright Data
- Oxylabs
- Smartproxy
- NetNut
- GeoNix
- IPBurger
- ProxySeller
- YouProxy

**核心机制**：
- **健康检查**：定期检查代理可用性和延迟
- **智能选择**：根据 API 类型、延迟、成功率选择最佳代理
- **使用统计**：记录每个代理的使用情况和成本
- **自动故障转移**：代理失败时自动切换到备用代理

### 6. 缓存模块 (`cache/`)

**职责**：
- 缓存 CDN 响应
- 缓存键生成和管理
- TTL 管理

**实现方式**：
- **Redis 缓存**：使用 Redis 存储缓存条目
- **缓存键格式**：`cache:v3:{sha256(method + url)}`
- **缓存内容**：状态码、响应头、响应体、内容类型

**缓存策略**：
- 只缓存 GET/HEAD 请求
- 大文件（>1MB）不缓存，直接流式传输
- 视频/音频文件可以缓存（最大 100MB）
- 默认 TTL：12 小时

### 7. 存储模块 (`storage/`)

**职责**：
- 白名单管理
- 配置管理
- 访问计数器
- 会话管理

**存储实现**：
- **Redis 存储**：使用 Redis 存储所有配置和状态
- **文件存储**：可选的文件存储实现（用于无 Redis 环境）

**数据结构**：
- **白名单**：Set 结构存储允许的域名后缀
- **配置**：Hash 结构存储系统配置
- **计数器**：Hash 结构存储每个域名的访问计数
- **会话**：Hash 结构存储管理后台会话

### 8. 管理后台模块 (`admin/`)

**职责**：
- Web 管理界面
- 白名单管理
- 配置管理
- 访问统计查看

**功能**：
- 登录认证（基于会话）
- 白名单增删改查
- 系统配置修改
- 访问统计查看

### 9. WebP 转换器 (`proxy/webp_converter.go`)

**职责**：
- 图片格式转换
- WebP 格式优化

**特性**：
- 并发限制（最多 5 个并发转换）
- 队列满时跳过转换
- Panic 恢复保护
- 可配置开关

## 数据流

### CDN 请求流程

```
客户端请求
    │
    ▼
HTTP 服务器 (main.go)
    │
    ▼
路由分发 (/path → Handler)
    │
    ▼
访问控制检查
    │
    ├─→ 拒绝 → 返回 403
    │
    └─→ 允许
         │
         ▼
    缓存查询 (Redis)
         │
         ├─→ 缓存命中 → 返回缓存响应
         │
         └─→ 缓存未命中
              │
              ▼
          请求上游 CDN
              │
              ▼
          响应处理
              │
              ├─→ 大文件 → 流式传输（不缓存）
              │
              └─→ 小文件 → 缓存响应 → 返回客户端
```

### API 请求流程

```
客户端请求
    │
    ▼
HTTP 服务器 (main.go)
    │
    ▼
路由分发 (/api.domain.com → Handler)
    │
    ▼
API 类型检测
    │
    ▼
住宅 IP 代理管理器
    │
    ├─→ 有可用代理 → 使用住宅 IP 代理
    │                  │
    │                  ▼
    │              请求上游 API
    │                  │
    │                  ▼
    │              流式传输响应
    │
    └─→ 无可用代理 → 普通代理
                     │
                     ▼
                 请求上游 API
                     │
                     ▼
                 流式传输响应
```

### WebSocket 请求流程

```
客户端 WebSocket 请求
    │
    ▼
HTTP 服务器 (main.go)
    │
    ▼
路由分发 → API Proxy
    │
    ▼
WebSocket 升级检测
    │
    ▼
获取 WebSocket 信号量
    │
    ▼
建立上游 WebSocket 连接
    │
    ▼
双向数据转发（goroutine）
    │
    ├─→ 客户端 → 上游
    │
    └─→ 上游 → 客户端
```

## 性能优化

### 1. 并发控制

- **请求并发限制**：信号量限制最多 50 个并发请求
- **WebSocket 并发限制**：最多 10 个 WebSocket 连接
- **WebP 转换并发限制**：最多 5 个并发转换

### 2. 内存优化

- **内存池**：64KB 缓冲区池，减少内存分配
- **流式传输**：大文件不加载到内存，直接流式传输
- **读取限制**：限制最大读取大小，防止 OOM

### 3. 连接池优化

```go
Transport 配置：
- MaxIdleConns: 50
- MaxIdleConnsPerHost: 25
- MaxConnsPerHost: 100
- IdleConnTimeout: 30s
- TLSHandshakeTimeout: 10s
- ResponseHeaderTimeout: 60s
```

### 4. 缓存优化

- **智能缓存**：根据内容类型设置不同的缓存策略
- **大文件跳过**：>1MB 文件不缓存，直接流式传输
- **压缩优化**：自动 gzip 压缩响应

### 5. 异步处理

- **批量保存**：配置和计数器异步批量保存
- **后台清理**：定期清理过期缓存和配置
- **资源监控**：独立 goroutine 监控资源使用

### 6. 超时控制

- **请求超时**：防止僵尸连接
- **上游超时**：CDN 请求 3 分钟，API 请求 5 分钟
- **服务器超时**：读取/写入超时 10 分钟（支持长连接）

## 安全机制

### 1. 访问控制

**CDN 请求访问控制**：
1. 非常见浏览器 User-Agent 允许
2. Referer 为 IP 或 localhost 允许
3. Referer 域名后缀在白名单中允许

**API 请求**：
- 不受访问控制限制（由上游 API 服务控制）

### 2. 管理后台安全

- **会话认证**：基于 Redis 的会话管理
- **密码保护**：环境变量配置管理员密码
- **会话过期**：默认 24 小时过期

### 3. 资源保护

- **并发限制**：防止 DoS 攻击
- **超时控制**：防止资源耗尽
- **文件描述符限制**：系统级限制（4096）

### 4. Panic 恢复

- **全局 panic 恢复**：防止单个请求 panic 导致服务崩溃
- **局部 panic 恢复**：WebP 转换等关键操作的保护

### 5. 输入验证

- **URL 验证**：验证上游 URL 格式
- **路径验证**：防止路径遍历攻击
- **大小限制**：限制请求和响应大小

## 扩展性设计

### 1. 模块化设计

- **接口抽象**：缓存、存储都使用接口，易于替换实现
- **插件化**：住宅 IP 代理提供商使用注册表模式，易于扩展

### 2. 水平扩展

**当前限制**：
- 单机部署，缓存和配置存储在 Redis（可共享）
- 可以部署多个实例，共享 Redis

**扩展方案**：
- 使用 Redis Cluster 支持更大规模
- 使用负载均衡器分发请求
- 使用 CDN 缓存静态资源

### 3. 垂直扩展

- **资源监控**：实时监控资源使用情况
- **动态调整**：可以根据负载调整并发限制
- **优雅降级**：高负载时自动降级服务

### 4. 功能扩展

**易于扩展的功能**：
- 新的住宅 IP 代理提供商（实现 Provider 接口）
- 新的缓存后端（实现 CacheInterface）
- 新的存储后端（实现 Storage 接口）
- 新的 API 类型（在 detectAPIType 中添加）

### 5. 监控和可观测性

**内置监控**：
- `/healthz`：健康检查端点
- 资源监控：定期记录 goroutine、内存使用
- 请求日志：记录所有请求的详细信息

**可扩展的监控**：
- Prometheus metrics（已预留接口）
- 分布式追踪（可集成 OpenTelemetry）
- 日志聚合（可集成 ELK/Loki）

## 总结

CDNProxy 采用分层架构设计，核心特点：

1. **高可用性**：全局 panic 恢复、优雅关闭、资源保护
2. **高性能**：并发控制、内存池、连接池、智能缓存
3. **可扩展性**：模块化设计、接口抽象、插件化架构
4. **易维护性**：清晰的代码结构、完善的日志、资源监控
5. **安全性**：访问控制、会话管理、输入验证、资源保护

该架构设计充分考虑了生产环境的需求，在性能、稳定性和可扩展性之间取得了良好的平衡。
